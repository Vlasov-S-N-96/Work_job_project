  3. Эквивалентное разбиение и анализ граничных значений

    4. Требование с валидациями для полей "Логин" и "Пароль" на странице регистрации. Оба поля принимают не просто буквы, а латиницу.

    • Необходимо заполнить таблицу тестовыми данными для проверки полей "Логин" и "Пароль" с учетом создания классов эквивалентности и граничных значений.

  4. Тестовые данные для фильтрации и сортировки

    5. Требования к фильтрации и сортировке:

    • Создать таблицу с исходными данными для попарного тестирования, включающую параметры и значения.
    • Сгенерировать тестовые данные для тестирования на основе алгоритма Pairwise.
    • Создать таблицу с двумя вкладками: исходные данные и тестируемые значения после применения алгоритма

    📌 Пример решения

  5. Тестовые данные для оплаты

    6. Необходимо применить технику тест-дизайна "Таблица принятия решений". В нашем приложении есть модуль, включающий оплату через Paypal.

    У аккаунта Paypal есть три статуса: valid, invalid и blocked, а также такая характеристика, как доступный баланс, который может быть нулевым и положительным (= достаточным для оплаты).

    В случае, если статус карты valid, а баланс на аккаунте достаточный для покупки, пользователь может совершить покупку. Во всех остальных случаях транзакция будет отклонена.

    • Создать таблицу, которая будет учитывать эти условия и выводить действие, согласно им.

    📌 Пример решения

  6. Создание чек-листа

    7. Ссылка на само приложение https://intern.demoshopping.ru/

    • Написать чек-лист, который будет включать тестирование по всем историям.

    📌 Пример решения

  7. Создание тест-кейсов
    • Создать не менее 10 тест-кейсов для всех историй.
    • Заголовок тест-кейса должен быть понятным.
    • У каждого шага должен быть ожидаемый результат.

    📌 Пример решения

  8. Поиск багов
    • Если вы нашли дефекты в ходе прохождения проверок из чек-листа, то создайте дефект и добавьте его ID в атрибут "BUG ID" из чек-листа.
    • Добавить не менее 5 отчетов.
    • Заголовок тест-кейса должен быть понятным.
    • По ходу выполнения тест-кейса необходимо создать дефект, если он есть. Выполнять дефекты через тестовые прогоны (Test Runs) в системах по управлению тест-кейсами.
